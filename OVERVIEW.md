# OVERVIEW

Chinese version: [OVERVIEW.zh-CN.md](./OVERVIEW.zh-CN.md)

## What this repository is

`@heyintech/sdkr` is a Nuxt module that turns filesystem-defined server APIs
into a typed client calling surface named `calla(...)`.

The current design has four core pieces:

1. Resolve one or more API collections from `nuxt.config`.
2. Scan handler files under each collection and convert file paths into Nitro
   route metadata.
3. Read exported `CallaMeta` types from handlers and generate one module type
   file, `types/sdkr.d.ts`.
4. Rewrite source-level `calla(...)` calls to `useFetch(...)` at build time.

This repository already has the main feature path working end to end. It is
still rough around documentation and test depth, but the architecture is no
longer in starter-template shape.

It is now organized as a `pnpm` workspace monorepo:

- `packages/nuxt` is the published package, `@heyintech/sdkr`
- `packages/cli` contains the CLI source and scaffold helpers
- `packages/nuxt/playground` is the local Nuxt verification app

Build output is bundled with `tsdown`, while Nuxt package declaration files are
generated by `scripts/build-nuxt-dts.mjs` to avoid over-bundling Nuxt ecosystem
types into the published `.d.ts` surface.

## Current model

Two distinctions are important:

- `calla(...)` is a required compile-time macro surface.
  The exported function in `packages/nuxt/src/calla.ts` is a type shell only. Real runtime
  behavior depends on the Vite transform rewriting calls to `useFetch(...)`.
- `defineApiCollection(...)` is an authoring helper, not a required macro.
  In normal execution it is just an identity function used for type checking.
  A standalone helper rewrite exists in `packages/nuxt/src/transform.ts`, but correctness of
  the module does not depend on erasing it.

## Module setup flow

The module entry is `packages/nuxt/src/module.ts`.

`ModuleOptions`:

- `collections: string[]`
- `injectCallaToGlobal: boolean`

During setup, the module does the following:

1. Resolves collection references from the consuming project's `rootDir`.
2. Imports each collection with `jiti`.
3. Derives `moduleRoot`, `serverRoot`, and `storesRoot`.
4. Watches each collection's `runtime/server` directory.
5. Scans route handlers and rejects conflicting routes.
6. Registers handlers with `addServerHandler`.
7. Aliases `#sdkr/calla` to the local `packages/nuxt/src/calla.ts` entry.
8. Emits one generated type file: `types/sdkr.d.ts`.
9. Adds `runtime/stores` directories to Nuxt auto-import scanning.
10. Installs the `calla(...)` to `useFetch(...)` Vite transform.

## Collection references

`packages/nuxt/src/collection-ref.ts` resolves the values inside `sdkr.collections`.

Behavior:

- References are resolved relative to the consuming Nuxt app's `rootDir`.
- If a ref does not explicitly point to a TypeScript file, `sdkr` treats it as
  a collection directory and appends `/collection.ts`.
- Glob patterns are supported before loading collections.
- Exact file paths still fall back to Nuxt `resolvePath` when needed.
- Duplicate matches are deduplicated.

Examples that work today:

- `modules/demo-api`
- `modules/demo-explicit/demo-module.ts`
- `modules/*`

## Collection contract

The user-facing contract lives in `packages/nuxt/src/collection.ts`.

`ApiCollection` fields:

- `name`
- `root?`
- `routeGroups`
- `ignore?`

Each `routeGroup` contains:

- `dir: string | RegExp`
- `clientPrefix: string`

Important details:

- `root` is optional.
  If omitted, `sdkr` uses the directory containing the loaded collection file.
- If `root` is provided, it may be a `file:` URL, an absolute path, or a path
  relative to the collection file.
- `defineApiCollection()` currently returns the input unchanged.

## Route scanning

`packages/nuxt/src/scan.ts` converts handler files into Nitro-compatible route registrations.

Discovery:

- Uses `tinyglobby`
- Scans `**/*.{js,mjs,cjs,ts,mts,cts,tsx,jsx}`

Path normalization rules:

- Strip file extensions.
- Remove route groups like `(group)/`.
- Convert `[id]` to `:id`.
- Convert `[...slug]` to `**:slug`.
- Convert `[...]` to `**`.
- Collapse `/index` to the parent route.

Suffix handling:

- `.get`, `.post`, `.patch`, etc. become HTTP methods.
- `.dev`, `.prod`, `.prerender` become Nitro env flags.

Output shape:

- `handler`
- `route`
- optional `method`
- optional `env`
- `lazy: true`
- `middleware: false`

Conflict detection in `packages/nuxt/src/module.ts` treats two handlers as conflicting when:

- computed `route` is equal, and
- either one has no method, or both methods are the same

## Type extraction and generation

The type pipeline is split between `packages/nuxt/src/calla-meta.ts` and `packages/nuxt/src/templates.ts`.

### `packages/nuxt/src/calla-meta.ts`

This file:

- parses handler source with `oxc-parser`
- looks for a named exported `CallaMeta`
- supports:
  - `export interface CallaMeta { ... }`
  - `export type CallaMeta = { ... }`
- extracts only three fields:
  - `body`
  - `query`
  - `res`
- caches parse work per file during one generation cycle

If parsing fails or no usable `CallaMeta` is exported, the handler is skipped
for type generation.

### `packages/nuxt/src/templates.ts`

This file builds the generated payload map and emits one Nuxt type template:

- `types/sdkr.d.ts`

That generated file contains:

- `declare module '#sdkr/calla' { interface InternalApiPayload ... }`
- optional `declare global { var calla ... }`

Generation rules:

- GET or methodless handlers fill both the route `default` schema and the
  explicit `get` schema.
- Method-specific handlers fill only their own method entry.
- Each schema may contain `body`, `query`, and `res`, depending on which fields
  were found in `CallaMeta`.

## `calla` type surface

`packages/nuxt/src/calla.ts` defines the type-level public API.

Main exports:

- `InternalApiPayload`
- `CallaQuery`
- `CallaBody`
- `CallaRes`
- `CallaOpts`
- `calla(...)`

Important details:

- `InternalApiPayload` starts empty and is meant to be augmented by the
  generated `sdkr.d.ts`.
- The generic structure is intentionally aligned with Nuxt `useFetch`.
- The sentinel generic uses `ResT = void`, matching Nuxt's current `useFetch`
  typing behavior.
- The function body is intentionally empty because source calls are rewritten
  away during build.

## Transform layer

`packages/nuxt/src/transform.ts` contains two independent pieces of transform logic.

### 1. `createCallaToUseFetchPlugin()`

This is the active module plugin.

Behavior:

- runs as a Vite `pre` plugin
- skips virtual modules, `node_modules`, `.nuxt`, and `.output`
- supports plain script files and Vue SFCs
- parses JS/TS with `oxc-parser`
- tracks lexical bindings and nested scopes
- rewrites only unshadowed identifier calls named `calla`

The scope tracking is deliberate. If user code defines its own local `calla`,
that symbol must not be rewritten.

### 2. `rewriteDefineApiCollectionMacro()`

This is a standalone utility, not part of module setup.

It can erase:

- `export default defineApiCollection({...})`

into:

- `export default {...}`

This is treated as an optional output optimization, not a correctness
requirement.

## Playground coverage

The local playground currently demonstrates two collection resolution modes.

Configured in `packages/nuxt/playground/nuxt.config.ts`:

- directory-like collection ref: `modules/demo-api`
- explicit TypeScript file ref: `modules/demo-explicit/demo-module.ts`

The playground page in `packages/nuxt/playground/app.vue` exercises:

- `calla('/demo/profile', { query })`
- `calla('/demo/echo', { method: 'post', body })`
- `calla('/demo-explicit/status', { query })`

This verifies:

- route scanning
- generated payload typings
- global `calla`
- the `calla` to `useFetch` rewrite
- both collection reference resolution modes

## Current rough edges

- Test coverage is still selective rather than comprehensive.
- Nuxt package declarations currently rely on a small custom emit script rather
  than `tsdown`'s built-in dts output, because the default dts pipeline was
  pulling large parts of the Nuxt type graph into the package surface.
- End-to-end tests that boot a Nuxt server may fail in restricted sandbox
  environments because ephemeral port allocation is environment-dependent.
- Some source files intentionally keep pragmatic loose AST typing because the
  project works directly against parser output rather than a fully modeled AST
  wrapper layer.
